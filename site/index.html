What is this, in plain English?

A working demo of the data platform behind a logistics control tower.

Trucks, parcels, and delivery riders constantly throw off events — GPS pings, parcel scans, delivery attempts. This project ingests those raw events through a streaming and batch pipeline, models them into a tested warehouse with dbt, and turns them into the numbers an operations team actually checks every morning: what’s running late, what’s stuck between hubs, and which delivery agents are overloaded. The dashboard further down runs on that exact pipeline — every number is computed from the data, nothing here is a mockup.

How it works

From raw events to operating metrics.

Five surfaces, one contract. Each step has tests, each layer is queryable, and the dashboard only reads from the warehouse.

1 Simulators GPS, scans, and delivery events synthesized at production-realistic volumes.
2 Kafka + Bronze Streaming ingest into immutable Bronze parquet via Spark Structured Streaming.
3 Silver jobs Trip, journey, and shift reconstruction — idempotent, schema-tested.
4 dbt + DuckDB Dimensional marts (fct_trips, fct_shipments, fct_agent_daily) with assertions.
5 Streamlit Read-only operator view served from the warehouse, deployed via Docker.

Streaming & batch

Kafka topics fed by simulators, consumed by Spark Structured Streaming into Bronze parquet; batch jobs reconstruct trips, journeys, and shifts with windowed logic and replay safety.

Dimensional modeling

dbt models split into staging, intermediate, and mart layers. SLA flags, productivity flags, and time dimensions land in fact tables consumed directly by the dashboard.

Data quality & ops

Schema, freshness, and business-rule checks emit JSON quality artifacts on every run. Docker, Compose, and CI verify the demo path end-to-end before any change ships.
